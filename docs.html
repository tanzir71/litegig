Quick start

Use a PHP 8 build with pdo_sqlite enabled. Runtime data is intentionally ignored by Git: local env files, SQLite databases, logs, and upload folders.

cp .env.example .env
php -S 127.0.0.1:8080

Open http://127.0.0.1:8080/litegig.php, register the first user, and that account becomes admin automatically.

The current local workspace may not have PHP on PATH. Run syntax and runtime checks on a host or container with PHP installed.

Demo accounts

After creating the first admin account, use Load Sample in the app nav. LiteGig inserts sample requester and runner accounts plus local gig requests across multiple task types.

Requesterrequester@example.test. Password is generated and shown after sample load.

Use a private test environment for sample credentials. If the users already exist, the app keeps the existing passwords and tells you so.

Core workflow

LiteGig models a local request as a custody path. The detail page shows who created the work, who currently owns the next operational move, and what action is available next.

Payment remains peer-to-peer. LiteGig only records manual confirmations and does not process or store payment details.

Deployment model

Keep hosting responsibilities separated:

  • Static repository pages: serve index.html and docs.html as public landing and documentation pages.
  • Vercel demo: serve vercel-demo/ as a static workflow preview that links back to the GitHub landing page.
  • Interactive app host: serve litegig.php on PHP 8+ with pdo_sqlite, sqlite3, and fileinfo.
  • Shared-host bundle: copy the app, env example, docs pages, and security tools to a PHP-capable folder such as public_html/litegig/.

Static hosts can publish the landing/docs layer, but they will not execute the PHP app unless the platform is explicitly configured for PHP runtime support.

chmod 750 litegig_data
chmod 750 ../litegig_uploads
chmod 640 .env
chmod 644 litegig.php index.html docs.html

Security expectations

The app includes output escaping, CSRF checks for state-changing forms, prepared SQL statements, session regeneration, private attachment downloads, upload validation, rate limits, CSP/security headers, and env-based runtime config.

  • Keep .env, databases, logs, and uploads outside public links.
  • Set a long random LITEGIG_CRON_TOKEN before enabling HTTP cron access.
  • Force HTTPS at the domain level and keep PHP/cPanel extensions patched.
  • Back up the database and upload directory daily before using real customer data.
  • Move from SQLite to a managed database when write concurrency grows.

Troubleshooting

SQLite driver missing

Enable pdo_sqlite in the PHP extension manager or deploy to a PHP host that supports SQLite.

Uploads fail

Check LITEGIG_UPLOAD_DIR, directory permissions, file size limits, and allowed file extensions.

Sample data cannot load

Confirm you are logged in as an admin. The first registered user is admin; later users are not.

CSV excludes emails

That is the safe default. Admin PII export requires the env flag to be enabled and an explicit export request.

Roadmap

  • Keep the static Vercel demo aligned with the PHP app workflow.
  • Add richer account switching for public demos without weakening production auth.
  • Introduce database adapters for MySQL or Postgres when SQLite is no longer enough.
  • Add notification hooks for status changes and comments.
  • Expand automated tests on a PHP-enabled CI runner.
