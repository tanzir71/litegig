Fast start

Pick the artifact for the host. Do not send the shared-host PHP zip to Vercel, and do not treat the Vercel zip as the PHP backend package.

1. Shared hostingRun npm run package:shared, upload dist/litegig-shared-*.zip, extract it, and open /health.php.
2. VercelRun npm run package:vercel, extract dist/litegig-vercel.zip, import the extracted folder in Vercel, and deploy.
3. Maintainer demoThe live demo is already deployed at vercel-demo-eight-sepia.vercel.app.
4. GitHub PagesOnly index.html, docs.html, static assets, and the landing surface belong here.

Deployment targets

Shared hosting / cPanelUse the PHP + SQLite zip. The host only needs PHP 8+, pdo_sqlite, sqlite3, and fileinfo.
VercelUse the separate Vercel artifact. It contains the Vercel-ready browser app fileset, not the PHP shared-host runtime.
VPS / SSHExtract the shared-host zip and sync the runtime folder to Apache, Nginx, or PHP-FPM.
GitHub PagesServe public marketing and docs only. Keep runtime code, data, and deployment zips excluded.

npm run package:shared
  • Upload dist/litegig-shared-*.zip to the domain or subdomain folder.
  • Extract it so litegig.php, health.php, .env.example, app/, brand/, styles/, and tools/ are in the app folder.
  • Open /health.php, then /litegig.php.
That is the shared-hosting path: use the generated zip. SSH, env edits, and cron setup are follow-up operations after the files are in place.

Vercel artifact

Vercel gets its own artifact. Do not upload dist/litegig-shared-*.zip to Vercel.

npm run package:vercel
Artifactdist/litegig-vercel.zip contains index.html, styles/tokens.css, vercel.json, package.json, and README.md.
Dashboard flowExtract the zip, put the contents in GitHub, import the repo in Vercel, choose Other or No Framework, and deploy.
CLI flowExtract the zip, enter the extracted folder, and run npx vercel@latest --prod.
Live demoThe maintained production demo is https://vercel-demo-eight-sepia.vercel.app.

The Vercel fileset is intentionally separate so Vercel deployment can evolve without dragging in PHP runtime assumptions or shared-host packaging rules.

First run

After the PHP zip is extracted on a PHP host, open /litegig.php and register the first real production admin. Then rotate or disable any seed/bootstrap accounts before real requests enter the queue.

Seed requesterrequester@example.test must not remain usable in production.
Seed runnerrunner@example.test must not remain usable in production.

The Admin Console prevents disabling the current active admin session and prevents disabling the last active admin. Every access change writes before/after snapshots to the audit log.

LLM deploy

Use this prompt when a deployment agent can read the repo or operate the hosting account.

Deploy LiteGig from this repo using the artifact that matches the host.
For shared hosting, use only dist/litegig-shared-*.zip and extract it on a PHP 8 host.
For Vercel, use only dist/litegig-vercel.zip and import the extracted fileset into Vercel.
Do not upload the PHP shared-host zip to Vercel.
Do not upload .git, dist history, local SQLite databases, logs, uploads, .env, or old zip files.
After a PHP deploy, verify /health.php, create a production admin, rotate seed access, and run a backup.

Security expectations

The PHP app includes escaped output, CSRF checks, prepared SQL statements, ownership checks, hashed delivery OTPs, private attachment downloads, upload validation, rate limits, CSP/security headers, health JSON, cron backups, and env-based runtime config.

  • Keep .env, databases, logs, and uploads outside public links.
  • Set a long random LITEGIG_CRON_TOKEN before enabling HTTP cron access.
  • Force HTTPS at the domain level and keep PHP/cPanel extensions patched.
  • Back up the SQLite database daily with php tools/maintenance.php --apply --backup.
  • Use provider feature flags for SMS/payment/error adapters; log-only/manual modes are valid low-infrastructure defaults.

Release and operations files: docs/release-checklist.md, docs/mysql-migration.md, docs/decisions/, and monitoring/uptime.example.yml.

Operations

  • Run php tools/maintenance.php --apply --backup from cron after the PHP app is live.
  • Run php tools/production_audit.php before handing a PHP deployment to operators.
  • Use npm run maintainer:deploy:demo only for the maintained Vercel demo project.
  • Use npm run package:vercel when a deployer needs a portable Vercel zip.

Troubleshooting

  • If /health.php fails, confirm PHP 8+, pdo_sqlite, sqlite3, fileinfo, and writable runtime directories.
  • If uploads fail, check LITEGIG_UPLOAD_DIR, directory permissions, file size limits, and allowed extensions.
  • If the Vercel import fails, confirm the extracted folder contains index.html, styles/, vercel.json, package.json, and README.md at the top level.
  • If GitHub Pages shows app files, confirm _config.yml excludes runtime folders, zips, data, and deployment-only files.
